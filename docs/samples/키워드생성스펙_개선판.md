## 1. 문제 정의 (원본 진단 유지)

레퍼런스 링크의 검색 키워드가 전 플랫폼에 동일하게 재사용되고 있다.
현재 코드는 Gemini가 섹션당 `searchQuery` 1개를 만들면 `buildPlatformQueries()`가
모든 플랫폼에 같은 검색어를 복사하는 구조다. 플랫폼마다 검색 모델·콘텐츠 유형·
유효 어휘가 다르므로, 생성 단계부터 플랫폼별 어휘로 갈라져야 한다.

- 나쁜 예 (Behance): `AI 빅데이터 지식 그래프 온톨로지 브로셔 사례 연구`
- 좋은 예: Behance `tech brochure design` / Dribbble `dark dashboard ui` /
  GDWEB `공공기관 홈페이지` / Mobbin `settings screen` — 같은 섹션이라도 플랫폼마다 다르게.

## 2. 구현 구조

### 2.1 플랫폼 프로필 (lib/reference/platforms.ts)

기존 `PlatformDef`(id·name·domains·mode·searchUrl — 레지스트리 필드는 그대로 유지)에
`keywordProfile`을 추가한다. **Gemini 프롬프트와 후처리 검증이 모두 이 프로필
하나를 참조한다** (규칙 이중화 금지).

```typescript
interface PlatformKeywordProfile {
  language: "ko" | "en";        // 쿼리 생성 언어
  maxWords: number;             // 쿼리당 최대 단어 수
  queriesPerSection: number;    // v1은 1 고정, v2에서 2로 확대
  vocabulary: string;           // Gemini 프롬프트에 삽입되는 어휘 지침 1문장
  goodExamples: string[];       // 실제 쿼리 문자열 2개 (프롬프트 예시)
  badPatterns: string[];        // 피해야 할 패턴 서술 (프롬프트 "금지 유형")
  strictVocabulary: boolean;    // true면 공용 금칙어(§4) 검증 적용
  fallbackQueries: string[];    // 검증 fail 시 폴백 (2개, v1은 [0]만 사용)
}
```

### 2.2 변환 위치: 프롬프트 생성 방식 채택 (유지)

- **채택:** `/api/section-queries` 프롬프트에 도메인 노출 플랫폼(`platformsForDomain(domain)`)의
  프로필(언어+maxWords+어휘 지침+good/bad)을 삽입해 처음부터 플랫폼별 쿼리를 생성시킨다.
- **기각:** 일반 키워드 생성 후 후처리 번역/치환 — 어휘 변환은 의미 재구성이라 후처리로는 품질이 안 나옴.
- **후처리의 역할은 검증만** (§4).

### 2.3 응답 스키마 (플랫폼 id 기준)

```typescript
// Gemini 응답 (섹션별)
{
  sectionId: string;
  layoutCandidates: string[];
  queriesByPlatform: { [platformId: string]: string };  // v1: 플랫폼당 1개
}
```

- 키는 플랫폼 **id**(`dribbble`, `gdweb` 등) — 표시명("Land-book")은 표기 흔들림이 있어 매칭 불안정.
- 기존 UI 타입(`ReferenceQuery.query: string`)·저장 JSON·복사 버튼·검색 URL 생성은 **무변경**.
  `SectionReference.searchQuery`(대표 검색어)는 유지하되, `platformQueries` 각 항목의
  `query`가 플랫폼별 생성값으로 채워진다.

### 2.4 병합 우선순위

Gemini 생성 쿼리(검증 통과) > 해당 플랫폼 `fallbackQueries`(폴백 전용).
둘을 섞지 않는다 — **플랫폼 단위 all-or-nothing**: 그 플랫폼 쿼리가 검증 fail이면
그 플랫폼은 폴백만 쓴다. (v2에서 2개로 확대돼도 같은 정책 — 1개만 fail해도 플랫폼 전체 폴백.)

## 4. 후처리 검증 (서버, 플랫폼별)

검증 순서 — 하나라도 fail이면 해당 플랫폼은 `fallbackQueries[0]`으로 교체:

1. **마스킹 토큰**: `[...]` 패턴 포함 시 제거 후 재검증 (기존 방어 유지)
2. **언어 일치**:
   - `ko` 프로필 → 한글이 **최소 1글자 이상** 포함되어야 통과 (영문 약어 혼용 정상: `CI BI 디자인`, `GNB 구성`)
   - `en` 프로필 → 한글이 포함되면 fail
3. **maxWords**: 공백 분리 단어 수가 프로필 `maxWords` 초과 시 fail
4. **금칙 패턴** (`strictVocabulary: true`인 프로필만): 공용 금칙어 목록 매칭 시 fail
   - 공용 금칙어(도메인 전문용어): 온톨로지, 지식 그래프, 파이프라인, API, 아키텍처,
     인프라, 온프레미스, 크롤링, 임베딩 등 — 분석 tags에서 넘어온 기술 키워드 유형
   - v1 등록 20종은 전부 `strictVocabulary: true`. 이 플래그는 후보 플랫폼(Google 등
     실서비스 검색형 — 기술 용어가 유효)을 위해 존재한다.

## 5. 완료 기준

### 자동 테스트 (lib/reference/platforms.test.ts 확장)

1. Behance/Dribbble은 en 프로필 — 한글 포함 쿼리가 fallback으로 교체됨
2. GDWEB은 ko 프로필 — 한글 없는 쿼리가 fallback으로 교체됨
3. 같은 섹션 입력에서 플랫폼별 쿼리가 서로 다름 (모의 응답 기준 매핑 검증)
4. 금칙어 포함 쿼리 / maxWords 초과 쿼리가 해당 플랫폼 fallback으로 교체됨

### 육안 스팟체크 (부니)

태안군 문서(모바일 앱/다크모드 케이스) 기준:
- Behance / Dribbble / GDWEB 3곳의 쿼리가 서로 **다른 어휘·다른 언어**로 생성되는지
- 생성 쿼리로 각 플랫폼 실검색 시 유의미한 결과가 나오는지

### v1 범위 확인

- 플랫폼당 쿼리 1개 생성
- 기존 `ReferenceQuery.query: string` 구조 유지
- UI·저장 JSON(재활용 모드) 호환성 유지 — 타입 변경 없음

## 6. 후보 플랫폼 (미등록 — 이번 구현 대상 아님)

감사 표에서 제안됐으나 현행 레지스트리에 없는 플랫폼. **등록하려면 별도 작업으로
domains(도메인 필터)·mode(auto-search/copy)·searchUrl 조사 + 프로필 정의가 선행돼야 한다.**
