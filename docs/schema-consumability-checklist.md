# 표준 디자인 MD 스키마 — 컨슈머빌리티 검증 체크리스트

> 목적: 표준 스키마를 확정하기 전에, A(Concept JSON)와 B(실측 화면) **양쪽 입력을
> 동시에** 표현할 수 있는지 검증한다. 한쪽만 보고 잡으면 추상화 수준 불일치로 재작업이 난다.
> 사용 시점: 표준 스키마 v0.x 초안을 잡은 직후, B 본격 구현 착수 전.

---

## 0. 검증에 쓸 샘플 (먼저 준비)

검증은 "머릿속"이 아니라 **실제 샘플을 펼쳐놓고** 해야 한다.

| 구분 | 샘플 | 개수 | 비고 |
| --- | --- | --- | --- |
| A 출력 | Concept JSON | 3개 | 웹/대시보드/문서형 각 1개 (성격 다른 것) |
| B 입력 | V0 HTML 화면 | 1~2개 | 실제 Vibe Coding 산출물 |
| B 입력 | Figma 파일 | 1개 | MCP로 읽을 수 있는 것 |
| B 입력 | 사내 기존 MD | 1개 (있으면) | 케이스2 검증용 |

샘플이 없으면 검증 자체가 추측이 된다. **A는 Phase 4까지 끝나야 진짜 Concept JSON이 나오므로, 이 체크리스트는 A Phase 4 완료가 전제다.**

---

## 1. 컬러 — 추상↔구체 양립 검증 (가장 위험한 지점)

A는 "Primary Blue 계열"처럼 방향으로, B는 `#196EFF`처럼 정확값으로 컬러를 가진다.
스키마가 둘 다 담을 수 있어야 한다.

- [ ] **C1.** primitive 토큰(실제 hex)과 semantic 토큰(역할)이 2계층으로 분리되어 있는가?
- [ ] **C2.** A Concept JSON의 "방향성 컬러"(예: 무드 기반 팔레트)가 스키마의 어느 필드에 들어가는가? 빈칸 없이 매핑되는가?
- [ ] **C3.** B 화면의 "실측 hex"가 primitive 토큰에 그대로 들어가는가?
- [ ] **C4.** A는 semantic만 있고 primitive가 비어있을 수 있다(컨셉 단계라 정확값 미정). 이때 스키마가 "primitive 미정" 상태를 표현할 수 있는가? (null 허용 or placeholder)
- [ ] **C5.** 반대로 B는 primitive는 꽉 찼는데 semantic 역할 매핑이 비어있을 수 있다. 이 상태도 유효한가?

> 핵심: A는 위(semantic)부터, B는 아래(primitive)부터 채운다. 스키마는 **한쪽만 채워진 상태도 유효**해야 한다.

---

## 2. 타이포그래피

- [ ] **T1.** A의 "타이포 방향"(예: 'Pretendard, 모던, 굵은 헤드라인')이 스키마 필드에 들어가는가?
- [ ] **T2.** B의 실측값(font-size px, weight, line-height)이 그대로 들어가는가?
- [ ] **T3.** 타입 스케일(h1~caption)이 역할 기반으로 정의되어, A의 추상 방향과 B의 실측값이 같은 슬롯에 수렴하는가?

---

## 3. Spacing / Radius / Elevation

- [ ] **S1.** B 실측 spacing scale(4/8/16/24...)이 토큰으로 들어가는가?
- [ ] **S2.** A는 spacing을 구체적으로 안 가질 수 있다. 이때 "스케일 미정" 또는 기본값 상속이 표현되는가?
- [ ] **S3.** radius/elevation도 같은 방식으로 A 미정 / B 실측 양쪽이 유효한가?

---

## 4. 컴포넌트 — granularity(상세도) 불일치 검증

A는 "카드형 레이아웃" 수준, B는 "button의 hover/pressed/disabled state"까지 가진다.
이 상세도 격차가 스키마에서 깨지지 않는지 본다.

- [ ] **M1.** 컴포넌트 섹션이 "있음/없음"이 아니라 **상세도 레벨**을 표현할 수 있는가? (예: 컨셉 레벨 / 구현 레벨)
- [ ] **M2.** A의 "버튼이 있다, 둥근 모서리" 수준이 들어가는가?
- [ ] **M3.** B의 "button: default/hover/pressed/disabled 각 상태별 스타일"이 들어가는가?
- [ ] **M4.** A 수준으로만 채워진 컴포넌트도 유효한 스키마 인스턴스인가? (state 없이도 OK)
- [ ] **M5.** 컴포넌트 범위가 정의되어 있는가? (button/input/card까지인가, 어디까지가 표준인가)

---

## 5. rules 섹션 — apply 스킬이 실행 가능한가 (B 핵심)

apply 스킬이 이 rules를 읽고 실제 화면에 전파한다. 사람이 읽는 설명이 아니라 **기계가 실행할 규칙**이어야 한다.

- [ ] **R1.** rules가 "치환 규칙"으로 구조화되어 있는가? (예: "모든 #XXX → semantic.primary로 치환")
- [ ] **R2.** apply가 컬러 1개 규칙만으로 dry-run 했을 때, 실제 화면 1개에서 치환이 동작하는가? (최소 파싱 테스트)
- [ ] **R3.** rules에 confidence/우선순위 개념이 있는가? (충돌 시 무엇을 우선)
- [ ] **R4.** A 컨셉 기반 MD에는 rules가 약하거나 비어있을 수 있다(추상이라). 이때 apply가 "rules 없음"을 어떻게 처리하는가?

---

## 6. 출력 포맷 (MD 본문 + JSON 블록 이중)

- [ ] **F1.** MD 본문(사람 검토용)과 후미 JSON(기계 파싱용)이 같은 내용을 담는가?
- [ ] **F2.** JSON 블록만 파싱해서 apply가 동작하는가? (MD 본문 파싱 의존 X)
- [ ] **F3.** A 변환기(Phase 5)가 Concept JSON → 이 출력 포맷으로 변환 가능한가? (필드 1:1 매핑표 작성)

---

## 7. 종합 판정

- [ ] **최종.** 위 항목 중 "A는 되는데 B가 안 됨" 또는 "B는 되는데 A가 안 됨"이 **하나도 없는가?**
  - 있으면 → 그 필드는 추상화 수준을 양립시키도록 스키마 보정.
  - 없으면 → 스키마 v1.0 확정, B 본격 구현 착수.

> 가장 흔한 실패: "A 기준으로만 잡아서 B의 실측 상세(컴포넌트 state, 정확 hex)가 들어갈 자리가 없음."
> 이 체크리스트는 그걸 착수 전에 잡기 위한 것이다.
