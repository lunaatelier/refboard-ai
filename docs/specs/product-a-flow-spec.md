# 제품 A — 단계별 정보 흐름 명세 (Phase 2~4 UX)

> 상위 컨텍스트: `CLAUDE.md`, 데이터 모델: `data-model.md`.
> 이 문서는 각 단계가 "무슨 정보를 / 어떻게 표출하고 / 사용자가 뭘 판단해 / 다음으로 뭘 넘기는지"를 정의한다.
> 원칙: 모든 단계는 **자동 생성(초안) → 사용자 선택·수정 → 확정** 3박자. AI 결과는 초안이며 사람이 확정한다(보안 KPI).

```
① 업로드 → ② 마스킹 → ③ 분석 → ④ 레퍼런스·무드 → ⑤⑥ 컨셉 3안+컨셉서 → ⑦ 디자인 MD(별도)
```

LNB 스텝퍼(사용자 언어): 업로드 / 마스킹 검수 / 분석 결과 / 레퍼런스·무드 / 컨셉 3안 / 디자인 MD.
각 단계: 완료✓ / 현재● / 잠김. 잠긴 단계는 `canAccessStep` 가드로 차단.

## ① 업로드

**Phase 1 File Upload Complete 입력 (고정, Step 4=txt/md만 → Step 5=+pdf/pptx):**
- 파일: txt / md / pdf / pptx (텍스트만)

**Phase 1.5 확장 입력:**
- 단일 이미지 : png / jpg / jpeg / gif — **구현됨 (Step 16).** 텍스트가 없으므로 고정 플레이스홀더가 maskedText 역할(마스킹 게이트 통과), 이미지 자체는 Step 9 opt-in 동의·재마스킹 경로를 그대로 탄다. 이미지 바이트는 브라우저 메모리에만 있고 자사 서버에도 올라가지 않는다. 이미지 분석 요약이 없으면 분석 시작 버튼이 잠긴다.
- 캡처 붙여넣기 (클립보드) — **구현됨 (Step 16).** 랜딩 화면에서 Ctrl+V. 클립보드의 이미지 항목만 수용(텍스트 붙여넣기 무시), 이후 단일 이미지와 동일 경로.
- 바이브코딩 링크 (V0 등) : 링크 입력 — **구현됨 (Step 17).** 공개 http(s) 링크만 지원. 서버(`/api/fetch-link`)가 정적 HTML 텍스트를 추출해 반환하며 **URL·본문을 로그·디스크에 남기지 않는다**(메모리 처리). 사설망·로컬 주소는 SSRF 방지를 위해 차단(2MB·10초 제한). 추출 텍스트는 파일 업로드와 동일하게 원문(parsedText) 취급 → 마스킹 게이트 필수 통과. 스크립트 렌더링 페이지(V0 미리보기 등)는 텍스트가 안 잡힐 수 있어 캡처 붙여넣기(Step 16)로 폴백 안내.

**이미지 포함 기획서 처리 (opt-in, 기본값 텍스트만 — Step 9):**
- **기본값 = 텍스트만 분석.** 이미지는 명시적 동의 없이 전송하지 않는다.
- pdf/pptx 안에 이미지가 있거나 이미지 업로드 시 → 경고:
  > "이미지 분석을 위해 Gemini로 전송됩니다. 무료 Gemini는 데이터를 학습에 쓸 수 있으니, 민감한 이미지가 있으면 제외해 주세요."
- 버튼: [이미지 분석 진행] / [이미지 제외, 텍스트만(기본)] / [취소]
- 진행 시: **전송될 이미지 목록 + 썸네일 + 제외 체크 + 민감도 힌트** 표시 → 동의분만 Gemini 멀티모달.
- **응답 재마스킹(필수):** 멀티모달 응답이 실명(로고·회사명)을 다시 뱉을 수 있으므로 저장 전 마스킹 엔진 한 번 더 통과. 저장은 항상 masked 기준.
- **OCR 선마스킹 (Step 18):** 동의 패널의 "전송 전 로컬 OCR 검사" 버튼 — 브라우저에서 이미지 텍스트를 읽어 마스킹 엔진으로 검사. 민감어 감지 시 해당 이미지 자동 체크 해제(기본 제외) + "OCR 민감어: 회사명 2 · 이메일 1" 형태 배지. OCR 원문은 즉시 폐기(종류·개수 요약만 유지), 미감지/글자없음도 배지로 표시.
- 텍스트 파싱은 기존대로(txt/md 브라우저, pdf/pptx 서버). 이미지 분석만 별도 동의.

**문서 타입 판정 (실사용#14) — 마스킹 전에 먼저 판정:**
```
업로드 직후, 텍스트 추출 결과로 문서 성격을 먼저 분류:
  documentPurpose:
    "project-brief"       기획서/화면설계서/화면정의서 (기본, 지금까지의 흐름)
    "company-profile"     회사소개서 (자사 마케팅 자료 — 조직도/재무/인증/수상)
    "template-only"       표지·간지·목차만 있는 템플릿 요청 (본문 없음)
```
- `company-profile`로 판정되면 경고: "이 문서는 회사소개서로 보입니다. 프로젝트 기획서와 함께 분석하면 도메인 판정이 왜곡될 수 있습니다." → 별도로 처리할지 확인.
- `template-only`로 판정되면 **경량 경로(실사용#22)** 안내: "본문 없이 표지·목차 구조만 있습니다. 전체 분석(③~⑤) 없이 표지·간지·목차만 바로 만들 수 있습니다."
  ```
  경량 경로: ① 업로드 → ② 마스킹(사업명만) → 이미지 힌트(사업명 기반 생성) → 표지/간지/목차 렌더
  → Phase 3(레퍼런스 전체)·Phase 4(컨셉 3안) 건너뜀. 출력 프리셋 중 "요약형"보다도 가벼운 별도 경로.
  ```
- 판정은 참고용 제안이며, 사용자가 "일반 경로로 계속 진행"을 선택하면 무시하고 기존 흐름대로.

## ② 마스킹 검수

**표출:** 초기 검증 단계에서만 화면 표출. 안정화 후에는 백단 자동 처리(테스트마다 화면 X).

**엔티티 민감도 등급 태깅 (Step 4는 2분, Step 6에서 6종):**
```
탐지된 회사명/기관명:
  가상물류    [✓ 가림]  등급: customer      ← 우리 고객사(기밀)
  가상커머스  [ 유지]   등급: competitor    ← 분석 대상 브랜드(공개) → 실명 유지
  가상투자사  [? ]      등급: investor      ← 애매(기밀 투자금 vs 공개 홍보) → 사용자 확정
```
- **Step 4(MVP):** 회사명에 "가림(기본)" / "유지(분석 대상 브랜드·참고)" 2분 토글.
- **Step 6(확장):** 6종 등급 — customer/investor/partner(기밀 기본 가림), competitor/benchmarkBrand/roleModel/publicReference(공개, 유지 가능). 각 항목 사용자 확정.
- 기본 전부 "가림"(안전). "유지"로 확정한 공개 엔티티만 실명 전송(분석 필요). AI 자동 판단 금지.
- 유지 태깅된 것은 `extractedAnalysisTargets`로 수집 → ④ "분석 대상 브랜드" 소스로 자동 연결.

**민감 수치 마스킹 (Step 6):**
- 회사소개서의 "누적 투자금 35억", "ARR 12억", "고객 43곳"은 개인정보는 아니나 외부 유출 위험.
- 3모드: exact-mask(`[투자금A]`) / range-generalize(`수십억 원대`) / keep.
- 기본: 투자금·매출·ARR·영업이익=exact / 고객수·사용자수·성장률=range / 공개 확정 시만 keep.
- range-generalize가 핵심: 완전히 가리면 Gemini가 규모감을 못 읽어 분석이 납작해짐. "수십억 원대"면 기밀 지키며 맥락 유지.
- 주의: 숫자 지표는 후보 탐지. 최종 판단은 사용자 검수(정규식만으론 투자금/매출 구분 어려움).

**다음으로:** 마스킹된 텍스트(기밀 가림, 공개 엔티티 유지, 수치 처리) + extractedAnalysisTargets 목록.

## ③ 분석 결과

**표출 정보:** 프로젝트명, 한줄설명, 도메인(+신뢰도), 타겟유저, 태그, 프로젝트종류, 브랜드컬러, 구성 페이지 목록, 각 페이지의 후보 섹션.

**사용자 판단:**
- 도메인/프로젝트종류 셀렉트 수정 (신뢰도 낮으면 사용자 우선)
- 브랜드컬러 수정
- **페이지 최대 5개 선택** (3개만 진행도 OK)
- **추가 요청사항 입력란(선택, Step 15 확장):** 지시를 **여러 개** 등록할 수 있고, 지시마다 **적용 단계(scope)** 와 **중요도(priority)** 를 선택한다.
  - scope 미선택 = 전 단계 적용(기존 동작·저장된 분석 JSON과 호환). 선택 시 해당 단계 프롬프트에만 주입.
  - scope ↔ 프롬프트 호출처 매핑: `analysis`=/api/analyze · `reference`=/api/targets-list·target-analyze·section-queries · `mood`=/api/mood·image-hints · `concept`=/api/concept.
  - `output` scope는 예외 — 산출물 렌더링은 로컬 결정론적(프롬프트 없음)이므로 주입되지 않고, 컨셉 단계 상단에 "다운로드 전 확인용 리마인더"로만 표시된다.
- **섹션 병합/삭제/수정** → confirmed (AI가 나눈 섹션을 사람이 다듬음)
  - 병합: 두 섹션 합치기 / 삭제: 불필요 섹션 제거 / 수정: 이름·성격 변경
- **페이지 제외 시 사유 기록:** 체크 해제하면 드롭다운(민감/범위밖/중복·품질/직접입력). 이 사유는 **이후 프롬프트에 "참조 금지" 조건**으로 들어감(기록용 아님).

**PPT 슬라이드 계보·페이지 역할 (Step 7):**
- **슬라이드 수 ≠ 페이지 수:** 여러 슬라이드가 1페이지일 수도, 1슬라이드가 여러 섹션일 수도. Gemini가 내용 기반 재구성(1:1 매핑 금지).
- 각 Page/Section에 `sourceSlides`(어느 슬라이드에서 왔나) 기록 → "이 내용 어디서 왔어?" 추적.
- `pageRole`(cover/section-divider/content/case-study/metrics/team/appendix/contact) 판정 → ⑤ 대표 페이지 추천에 활용.

**제외 페이지 차단 (중요):**
- Phase 3 이후 프롬프트에는 selectedPages·confirmedSections만 넣는다.
- 제외 페이지는 "Excluded pages must not be used as source material. Reason: ..."로 명시 → AI가 이미 읽은 문맥으로 제외 내용을 은근히 반영하는 것 방지.

**도메인별 섹션 세트 (contentType / layoutPattern):**
| 도메인 | contentType 예 | layoutPattern 예 |
| --- | --- | --- |
| dashboard-ops | filter-bar, data-table, map-panel, chart-widget, stat-summary, detail-panel, list-feed | table, heatmap, line-chart, bar-chart, split-panel |
| document(문서형) | history, business-model, team, pricing, vision | timeline, comparison-table, card-grid, hero |
| marketing-web | hero, feature, testimonial, cta, gallery | hero-visual, card-grid, carousel, stat-band |
| mobile-app | onboarding, list, detail, profile, settings | list-view, card-stack, tab-bar, bottom-sheet |
- enum은 열어둠(`| string`) — 예외 섹션 수용. Gemini가 도메인 판정 후 해당 세트로 섹션 분류.
- 대시보드도 섹션이 명확히 나뉜다(필터바/테이블/지도패널 등). 문서형에 치우치지 않음.

**문서에 이미 있는 콘텐츠 변형 감지 (실사용#24 — 우선순위 최상):**
```
분석 중 다음 패턴을 감지:
  - 같은 화면/섹션이 "Main1/Main2/Main3"처럼 N개 변형으로 이미 존재
  - 각 변형이 카피·톤·구성 순서만 다르고 기능 요소는 동일
감지되면:
  existingContentVariants: { variantId, label, sourceSlides, contentSummary }[]
  → 사용자에게 "이 문서에 이미 3개 시안이 있습니다. 이걸 기반으로 컨셉 3안을 만들까요?" 확인
  → 확정하면 ⑤ 컨셉 3안 생성 시 AI가 처음부터 만들지 않고 이 변형을 ConceptOption에 1:1 매핑,
     그 위에 UI 톤(색감·레이아웃)만 옵션별로 다르게 얹는다.
  → 감지 안 되거나 사용자가 거부하면 기존대로 AI가 3안을 처음부터 생성.
```

**문서 내 기존 사례분석 재활용 (실사용#2):**
```
분석 중 "사례분석/벤치마킹" 성격의 섹션(경쟁사·레퍼런스 서비스에 대한 설명 + 이미지 + 출처)이
이미 문서에 있으면 감지하여:
  detectedCaseStudies: { name, sourceSlides, extractedNote, sourceUrls }[]
→ ④ 분석 대상 브랜드 목록(1단계)에 "이미 분석됨" 상태로 선반영 제안.
   Gemini grounding으로 처음부터 재조사하지 않고, 문서의 기존 분석을 검수 후 활용.
```

**표 단위 재무/수치 정보 (실사용#15):**
```
다년도 재무제표처럼 문장이 아니라 "표 전체"가 수치 덩어리인 경우:
  일반 문장 속 단일 수치(예: "누적 투자금 35억")는 문장 단위 NumericDetection 1건.
  표 형태(연도×항목 매트릭스)는 표를 파싱해 행/열 단위로 NumericDetection을 다건 생성.
  표 구조(연도·항목 라벨)는 유지하고 셀 값만 마스킹/범주화 — 표가 마스킹 후에도 읽을 수 있는 형태 유지.
```

**저장 & 재활용 (핵심 — Step 13):**
- 분석 완료 시 **"분석 결과 JSON"** 저장 가능(마스킹된 상태 = 실명 없음, 안전).
- 재활용: 이 JSON을 ①에 올리면 **마스킹·분석 건너뛰고 ④ 레퍼런스부터 재시작**(`sourceType: "analysis-json"`, 가드가 reference 접근 허용).
- 용도: 같은 분석으로 레퍼런스·무드를 다른 결과로 다시 보기 / 페이지 껐다 켜기 / 나중에 나머지 페이지 추가.
- JSON엔 전체 페이지가 저장됨 → 1차에 3개만 골랐어도 재활용 때 나머지 선택 가능.
- **재활용 모드 3구분 안내:** 일반(원본 시작, 실명본 가능) / 재활용(JSON 시작, 마스킹본만) / 같은세션 재활용(mappings 메모리 있으면 실명본 가능). ④ 진입 시 "이 모드는 마스킹본만 출력됩니다" 안내 + 다운로드 버튼에 마스킹본/실명본 가능 여부 표시.

## ④ 레퍼런스·무드 (+분석 대상 브랜드)

> ⚠️ 이 페이지는 정보량이 많다. **스크롤을 길게 늘어놓지 않는다** — 아래 "레이아웃 원칙" 준수.

### 레이아웃 원칙 (스크롤 최소화)
- **탭 또는 세그먼트로 분할:** [컬러·무드] [섹션별 레퍼런스] [분석 대상 브랜드] [이미지] 를 한 페이지에 세로로 쌓지 말고 탭 전환.
- **섹션별 레퍼런스는 아코디언:** 선택한 섹션들을 접힌 상태로 두고, 펼친 것만 상세 표출. 기본 1개만 펼침.
- **분석 대상 브랜드·이미지는 그리드 카드:** 세로 나열이 아니라 카드 그리드로 한눈에.
- **좌측 고정 요약 패널(선택):** 현재까지 확정한 팔레트·무드를 좌측에 고정, 우측만 스크롤.
- 원칙: "한 화면에서 파악, 깊이 보고 싶은 것만 펼침." 세로 무한 스크롤 금지.

### 레퍼런스 매칭 기준 — "생김새"가 아니라 "행위 성격" (실사용#31 — 우선순위 최상)
```
문제 사례: "관리자페이지" 문서인데 내용(GNB/LNB/컴포넌트 정책)이 대민 홈페이지를
          그대로 물려받아 작성된 경우, 화면 생김새만 보고 도메인을 판정하면
          엉뚱한 레퍼런스(공공기관 홈페이지 스타일)가 붙어버림.

해결: 도메인/레퍼런스 판정 기준을 "화면 시각 스타일"이 아니라
      "메뉴·기능의 행위 성격"으로 우선한다.
        게시판 관리·배너 관리·회원 관리 등 CRUD/콘텐츠관리 행위 → CMS 관리자 레퍼런스
        GIS 대시보드·통계 조회·필터·차트 등 분석/모니터링 행위 → 분석형 대시보드 레퍼런스

부모-자식 사이트 관계 인식:
  분석 단계에서 "이 문서가 다른 업로드/기존 프로젝트의 관리자 페이지인지" 후보를 제시.
  parentSiteRelation?: { relatedProjectId?: string; relationNote: string }
  관계가 확인되면 레퍼런스도 "일반 관리자 대시보드"가 아니라
  "이 부모 사이트를 관리하는 CMS 백오피스" 쪽으로 좁혀서 검색.
```

### 공개 마케팅 자료의 고객사명 — 가리면 안 됨 (실사용#18)
```
회사소개서처럼 "우리 회사가 스스로 공개한 자료"에 등장하는 고객사·파트너명
(예: "테슬라/BMW/벤츠 서비스 제공", "SKT 최우수파트너 수상")은
publicReference 등급으로 분류하되, 일반 publicReference와 결이 다르다:
  - 일반 publicReference: 문서 밖 제3자가 참고 대상으로 언급
  - 이 경우: 문서 작성 주체 스스로가 실적/신뢰도로 내세우는 대외공개 정보
→ 마스킹 검수에서 "회사소개서/마케팅 자료" 문서 타입(#14)으로 판정된 경우,
  회사명 탐지 규칙의 기본값을 "가림"에서 "유지 후보"로 전환(과잉마스킹 방지).
```

### 이메일 도메인도 회사 식별자 (실사용#11)
```
이메일 주소의 @ 뒤 도메인(예: shsong@mqnic.com의 "mqnic.com")도
회사명 탐지의 보조 신호로 사용한다. 이미 등록된 회사명과 도메인이 일치하면
같은 엔티티로 묶어 태깅 일관성을 유지(같은 회사인데 "엠큐닉"과 "mqnic.com"이
별개로 탐지되는 것 방지).
```

  Text(본문)         → [블랙 ▾]

  Navigation(LNB/GNB)→ [화이트 ▾]   ← "LNB 블루" 싫으면 변경
  ↓ 실시간 미리보기 (버튼/카드/LNB/본문 샘플에 즉시 반영)

- 스크린샷 안 씀(링크 방식) — 무료 스크린샷 할당량은 분석 대상 브랜드에 아낌.

### 분석 대상 브랜드 (경쟁사·롤모델·투자사 등 — 2단계 + 깊이)
**소스 3갈래:** 설계서 자동추출(②의 공개 엔티티 태깅) + 사용자 직접입력 + Gemini grounding 추천.
> "경쟁사"로 한정하지 않는다 — 벤치마킹 롤모델(예: 회사소개서의 "가상아웃도어처럼"), 공개 투자사도 같은 소스로 다룬다.

**2단계 구조 (넓게 → 깊게):**
```
1단계: 목록 넓게 15~20개
  이름 + 한줄 특징 + 가벼운 썸네일(favicon/메타이미지, 스크린샷 아님 — 할당량 절약)
  [+ 더 찾기]로 목록 확장(Gemini 추가/직접입력). 목록은 계속 유지, 사라지지 않음.
        ↓ 사용자가 훑고 선택
2단계: 고른 것만 깊게 분석 (병렬 요청 + 완료되는 대로 점진 표출)
  깊게(3~5개): 7개 축 분석
  아주 깊게(1~2개): + 스크린샷 멀티모달
```

**깊은 분석 = 7개 축 (구조화 질문으로 깊이 확보):**
"분석해줘"(얕음) 대신 아래 축을 각각 강제:
1. 레이아웃 전략 (정보구조, 네비 배치와 이유)
2. 컬러·비주얼 전략 (메인 컬러 의도, 다크/라이트 이유)
3. 컴포넌트 패턴 (데이터 표현: 테이블/카드/차트 비중)
4. 페인포인트 (이 디자인의 약점)
5. 와우포인트 (배울 점)
6. 추정 의도 (왜 이렇게 디자인했나 — 타겟·비즈니스 맥락)
7. 우리 프로젝트 시사점 (차용할 것 / 피할 것)
- 스크린샷을 멀티모달로 같이 넣으면 분석이 구체화됨(보여주기 + 분석입력 이중 역할).
- grounding으로 출처 URL 확보(환각 방지). 모든 결과에 "추정 포함, 확인 필요" 배지.

**분석됨 ≠ 채택:**
```
✓ 가상커머스    [분석됨]  ☐ 채택   ← 별로면 채택 안 함
✓ 네이버  [분석됨]  ☑ 채택   ← 맘에 들면 체크 → 컨셉으로
○ CJ      [대기]            ← 나중에 추가 분석 가능
```
- 분석 결과는 누적(사라지지 않음). 채택한 것만 ⑤ 컨셉으로. 0개 채택도 가능.

**저장 (2층):**
- 프로젝트 JSON에 분석 대상 브랜드 분석 저장(재활용 구조와 연결).
- 분석 대상 캐시: "가상커머스" 분석은 프로젝트 넘어 재활용. 재요청 시 "N일 전 분석" 표시 + [새로 분석] 옵션.
- 분석 대상 브랜드는 공개 정보라 실명 저장 OK(고객사 기밀 아님 — 마스킹 정책과 일관).

**스크린샷 처리:**
- 무료 스크린샷 API(할당량 내) → 소진 시 자동으로 링크+텍스트 분석 폴백.
- 분석 대상 브랜드 분석에만 사용(꼭 넣음). 레퍼런스 플랫폼엔 안 씀(할당량 부족 방지).

### 이미지 힌트 (Step 11: 프롬프트·타입·scale 표출 / Step 19: 실제 생성)
- 타입 판정 기준: **도메인 + 선택한 무드.**
  - 금융/B2B → 사진형 or 미니멀 3D / 이벤트·키즈 → 일러스트 2D / 혁신 무드 → 3D 렌더 / 미니멀 → 라인 일러스트
- **기본 표출 (Step 11, NVIDIA 없이 동작):**
  ```
  [로그인 이미지 힌트]
   타입: 3D 일러스트   scale: hero
   프롬프트: "abstract 3D geometric, blue gradient..."
  ```
  - scale(hero/section/icon) 구분 + 프롬프트 + 타입까지가 **기본이자 현재 구현 대상.**
- **실제 생성 (Step 19 구현됨 — NVIDIA NIM):**
  ```
  [로그인 이미지 힌트]
   타입: 3D 일러스트   프롬프트: "abstract 3D geometric, blue gradient..."
   [🎨 이미지 생성] → 생성 썸네일 표시 (hint.generatedImageUrl, data URL)
  ```
  - `/api/generate-image`: GET = 활성 여부(`NVIDIA_API_KEY` 존재), POST = 프롬프트→이미지. 서버에서 마스킹 토큰을 방어적으로 한 번 더 제거 후 전송.
  - aspectRatio("16:9" 등)를 64배수 픽셀로 변환(최대 변 1024). 기본 모델 `black-forest-labs/flux.1-dev`(실측 안정), `NVIDIA_IMAGE_MODEL`로 교체.
  - **키 미설정 = 에러 아님**: 생성 버튼 비활성 + 안내 툴팁, 프롬프트 복사 경로는 그대로 동작 (자동 폴백).
- 사용자가 일러스트/사진/3D 토글로 타입 변경 가능(Step 11부터 가능, 생성 여부와 무관).

**원본 이미지 참고 금지 플래그 (실사용#20 — 제안서 템플릿 케이스):**
```
문제 사례: 발주처가 준 제안서 템플릿 표지에 이미 배경 이미지가 있는데,
  "이 이미지는 참고하지 말고, 사업명 텍스트만으로 새로 구상해서
   레퍼런스 찾고 생성해달라"는 요청이 실무에서 자주 있음.

해결: 이미지 힌트 생성 시 명시적 플래그를 둔다.
  ImageHint.sourceReferenceMode: "use-source-image" | "text-only-ignore-source"
  - 기본값(use-source-image): 원본 이미지가 있으면 참고
  - text-only-ignore-source: 원본 이미지가 있어도 무시, 사업명 등 텍스트 시맨틱만으로
    프롬프트 생성. 문서 타입이 "template-only"(#14)로 판정되면 이 모드를 기본 제안.

사업명은 의미보존 마스킹 필수(실사용#23과 연결):
  기관명만 [기관A]로 가리고 "헬기 운항관리시스템 고도화" 같은 사업 성격 키워드는
  유지해야 이미지 프롬프트가 의미를 가짐. 사업명 전체를 통째로 토큰화하지 않는다.

  (실사용#26) 같은 원리가 화면 카피 자체에도 적용된다 — 예: "국가교통정보센터가
  국민 여러분의 힘찬 한해를 기원합니다" 같은 실제 UI 문구 안에 고객사명이 자연스럽게
  섞여 있으면, 분석용으로는 [기관A]로 가리되 최종 화면 카피 생성 시엔 복원해서
  실제 문구를 살린다. 분석용 마스킹과 최종 산출물 복원을 항상 구분한다.
```

## ⑤⑥ 컨셉 3안 + 컨셉서

**컨셉 방향:** ④에서 사용자가 선택한 결과물 기준으로 방향 잡음.

**플랫폼별 산출물 (실사용#25 — 웹+모바일 동시 요구):**
```
일부 프로젝트(계절성 캠페인 홈페이지 등)는 "웹 화면 + 모바일 화면"을
별도 산출물로 명시적으로 요구한다(반응형 자동변환이 아니라 각각 독립 설계).
ConceptOption.platforms?: { web?: ConceptPage[]; mobile?: ConceptPage[] }
  - 지정 없으면 기존처럼 웹 기준 단일 세트.
  - 웹+모바일 요구가 감지/지정되면 두 세트를 각각 3안으로 생성.
  - 모바일 세트는 웹의 축소가 아니라 독립적인 레이아웃(세로형·터치 UI 고려).
```

**구조 = 전체 방향 3안 (프로젝트 관통):**
```
컨셉 3안 = 프로젝트 전체를 관통하는 방향 3가지
  각 안 = UiStructure(전체) + KeyVisual(전체) + 3축 컨셉서 + 대표 페이지 시각화
  대표 페이지(메인) 1개로 각 안을 시각화 → 방향 확정 후 나머지 전개
```
- 5개 페이지 각각 3안(=15개) 안 만듦 — 결정 불가·시간 폭발. 전체 3안을 대표 페이지로 보여줌.
- **표지 ≠ 대표 (2종 분리):**
  - **시각 대표(visualRepresentativePageId):** 보통 표지(cover). 첫인상·키비주얼 톤을 보여줌. 컨셉서 맨 앞 슬롯에 고정.
  - **내용 대표(contentRepresentativePageId):** content/metrics 페이지. 정보구조·섹션배치·콘텐츠 매핑을 보여줌 → 3안의 진짜 차이가 드러남.
  - 대표 페이지 기본 추천은 **도메인별**: 문서형=사업영역/성과(content·metrics), 대시보드=메인 대시보드, 마케팅웹=랜딩 히어로, 앱=핵심기능 화면. cover는 내용 대표로 쓰지 않음(내용 빈약).
  - 둘 다 사용자 변경 가능.

**3안 차별화 축 (도메인별):**
| 도메인 | 3안을 뭘로 다르게 |
| --- | --- |
| UI(대시보드·앱) | 다크/라이트, GNB 위치(상단/좌측), 정보밀도(집약/여백) |
| 문서형 | 표지 톤(다크/라이트), 이미지 타입(일러스트/사진), 그리드 |
| 마케팅웹 | 무드(신뢰/혁신/미니멀), 레이아웃 대담함 |

**3안 모두 내보내기:** 1안 확정이 아니라 **3안 다 컨셉서(PPT/PDF)로.** 시안은 보통 3안 제출.

**출력 프리셋 (밀도 제어 — 볼륨 폭발 방지):**
| 프리셋 | 용도 | 포함 내용 |
| --- | --- | --- |
| 요약형 (summary) | 빠른 내부 공유 | 3안 비교 + 대표 페이지 + 핵심 키워드 |
| 제안형 (proposal) | 클라이언트 제출 | 3안 + 표지/대표/선택 서브 + 디자인 방향 |
| 상세형 (detailed) | 제작 전달 | 섹션별 콘텐츠 매핑 + 레퍼런스 근거 + 이미지 힌트까지 |
- 문서형에서 3안 × (메인+서브 4개)를 다 상세형으로 뽑으면 금방 비대해짐 → 프리셋으로 조절.

**컨셉서 구성 (표지 + 메인1 + 서브N 선택):**
```
표지: (visualRepresentativePage, 항상 맨 앞)
내용 대표: [사업영역 ▾]  ← content/metrics 중 (도메인 추천)
서브 페이지: ☑ 회원관리  ☑ GIS모니터링  ☐ 리포트  ☑ 설정  ← 선택한 것만 포함
→ PPT/PDF 자동 구성:
   0. 표지 (키비주얼 대표, 항상 포함)
   1. 전체 컨셉(관통 방향, 3축 키워드)
   2. 내용 대표: 섹션별 상세(레이아웃+키비주얼+콘텐츠 매핑)
   3~. 서브: 선택한 페이지별 섹션 설명(메인 방향 계승)
```
- 선택한 서브는 다 들어가되, 어떤 걸 넣을지는 체크박스로.
- 섹션별 콘텐츠 매핑(마스킹된 실제 본문)이 화면 영역에 들어감. 실명은 렌더 시점 복원.

**컨셉서 다운로드:** 이 단계 안에 포함(PPT/PDF). 여기까지가 제품 A "1차 완결".

## 부록 — API/기술 방침

- **호출 추상화:** LLM/이미지 생성 호출을 한 모듈로 모아둠 → 나중에 NVIDIA 등으로 교체 쉽게(OpenAI 호환).
- **텍스트 분석:** Gemini(`gemini-3-flash`, grounding 포함).
- **이미지 생성:** NVIDIA NIM 이미지 모델(무료, 개발용). 40 RPM·개발용 약관 유의 → 개인용 OK, 팀 배포 시 재검토.
- **스크린샷:** 무료 스크린샷 API(할당량) → 폴백. Playwright 자체 구축은 무겁고 배포 복잡 → 후순위.
- **NotebookLM:** 공식 API는 기업용, 비공식은 불안정 → 후순위(분석 대상 브랜드 분석은 Gemini grounding으로 충분).
- **rate limit 대응:** 분석 대상 브랜드 넓은 목록은 한 호출로 묶어 뽑기, 깊은 분석은 병렬+점진 표출, 분당 15 이내.
